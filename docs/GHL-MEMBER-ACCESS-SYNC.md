# GHL member access mirror

Gaia does not decide a member's tier or access. GHL remains the source of truth. The app stores only the access-granted/access-removed events needed to show the signed-in contact what GHL already allows.

## Receiver

`POST https://api.gaiahealers.app/api/webhooks/ghl/member-access`

Every GHL Workflow webhook must send this header:

```text
X-Webhook-Secret: <GHL_WORKFLOW_WEBHOOK_SECRET>
```

The endpoint is idempotent when `eventId` or `idempotencyKey` is supplied. It stores the GHL contact ID, explicit tier, tags, and granted course/community identifiers. It does not store member email addresses.

## Workflow payloads

Offer/Course access granted:

```json
{
  "event": "course_access_granted",
  "eventId": "<unique workflow execution/event ID>",
  "contactId": "<GHL contact ID>",
  "courseId": "<course or offer ID>",
  "courseName": "<course or offer name>",
  "openUrl": "<member-facing course URL>"
}
```

Offer/Course access removed uses the same body with `"event": "course_access_removed"`.

Group/Community access granted:

```json
{
  "event": "community_access_granted",
  "eventId": "<unique workflow execution/event ID>",
  "contactId": "<GHL contact ID>",
  "groupId": "<group/community ID>",
  "groupName": "<group/community name>",
  "openUrl": "<member-facing community URL>"
}
```

Group/Community access revoked uses the same body with `"event": "community_access_revoked"`.

An explicit membership workflow can send:

```json
{
  "event": "membership_tier_updated",
  "eventId": "<unique workflow execution/event ID>",
  "contactId": "<GHL contact ID>",
  "tier": "Free"
}
```

Allowed tier values are Free, Silver, Gold, and Diamond. Contact tags are also read live; payment amounts are never used to guess a tier.

## Existing-member backfill

Publishing workflows handles future grants and removals. Existing contacts must be enrolled once into a backfill workflow (or have their access re-granted) so GHL emits the same payloads. Backfill must use the actual offer/group IDs for each member; do not bulk-assign a tier or product in Gaia.

## GHL Workflow AI prompt

Use this prompt in Workflow AI, substituting the secret only in the secured header field:

> Create four contact workflows: Offer Access Granted, Offer Access Removed, Group Access Granted, and Group Access Revoked. For each trigger, send a POST webhook to https://api.gaiahealers.app/api/webhooks/ghl/member-access with Content-Type application/json and an X-Webhook-Secret header. Include the triggering contact ID, the offer/course or group/community ID and name, its member-facing URL when available, and a unique execution ID. Use event values course_access_granted, course_access_removed, community_access_granted, and community_access_revoked respectively. Do not grant access, change tags, change memberships, or send customer messages; only reflect the access event to Gaia.

Review the merge fields generated by Workflow AI before publishing. Trigger names and available merge fields can vary by GHL location and product configuration.
